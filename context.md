# DetectLLM Context

This repository implements the DetectLLM paper pipeline for zero-shot detection of machine-generated text.

## What the project does

- Generates paired text samples: `original` human text and `sampled` text produced by a chosen causal language model.
- Evaluates multiple detectors on those pairs, including likelihood-based, rank-based, perturbation-based, and supervised detector baselines.
- Saves all results as JSON under a timestamped `results/.../result.json` folder.

## Main execution flow

1. `main.py` parses CLI arguments for dataset, base model, perturbation settings, and baselines.
2. `baselines.utils.preprocessing.preprocess_and_save` creates the cache path and output folder path.
3. `baselines.utils.loadmodel` loads:
   - a causal base model + tokenizer for generation and scoring
   - a T5-style mask-filling model for perturbations
4. `baselines.sample_generate.generate.generate_data` loads a dataset, filters/truncates examples, and generates sampled continuations from the base model.
5. `baselines.all_baselines.run_all_baselines` computes detector scores and AUROC metrics.
6. Results are written to `result.json`.

## Core detector ideas

- `likelihood`: average log likelihood under the base model.
- `rank`: average token rank under the base model.
- `logrank`: average log token rank.
- `entropy`: token-level entropy of the base model predictions.
- `LRR`: log-likelihood divided by log-rank, intended as a fast zero-shot signal.
- `DetectGPT`: compares original text to perturbed variants using log-likelihood curvature.
- `NPR`: compares original text to perturbed variants using log-rank changes.
- `supervised_roberta_base` / `supervised_roberta_large`: OpenAI detector checkpoints.

## Data sources

- `xsum`, `squad`, `writing` are the main datasets exercised by `run.sh`.
- Extra dataset loaders exist for `english`, `german`, and `pubmed` in `baselines/sample_generate/custom_datasets.py`.

## Important implementation details

- Generation typically uses the first `prompt_len` tokens as context.
- `generate_data` filters examples to keep reasonably long sequences and to fit within the mask tokenizer limit.
- Perturbation methods rely on T5-style mask filling and can be much slower than the non-perturbation baselines.
- `run.sh` shows the intended experiments across several base LLMs and datasets.

## Practical notes

- The code is research-oriented and assumes access to large Hugging Face models.
- Some paths and defaults are tuned for the original paper experiments rather than lightweight local runs.
- `baselines/utils/run_baseline.py` also contains plotting helpers that appear to rely on names like `args`, `SAVE_FOLDER`, and `base_model_name` from outer scope, so those helpers are not fully self-contained.

## Research directions and possible extensions

These are the most concrete next directions for extending the research in a way that fits the current codebase.

### High-priority extensions

- Cross-model generalization: evaluate the detector on text generated by a different base model than the one used for scoring. This would test whether the signal is model-specific or more universal. The natural entry points are `main.py` and `baselines/sample_generate/generate.py`.
- Perturbation ablations: compare the current T5-based perturbations against random fills, random token replacement, or alternative mask-filling models. This would show whether the gain comes from the perturbation strategy itself or from T5 specifically. The main logic lives in `baselines/detectGPT.py`.
- Masking hyperparameter sweeps: vary `pct_words_masked`, `span_length`, and `mask_top_p` to check how robust DetectGPT and NPR are to perturbation settings. This also belongs in `baselines/detectGPT.py` and `main.py`.
- Token-position analysis: inspect whether early, middle, or late tokens contribute most to rank and likelihood separability instead of only averaging over the full sequence. The relevant code is in `baselines/rank.py` and `baselines/loss.py`.
- Length-stratified evaluation: report AUROC by text-length bins, since the current pipeline filters by length but does not analyze performance as a function of length. This can be added around `baselines/sample_generate/generate.py` and `baselines/all_baselines.py`.

### Additional useful extensions

- Statistical reliability: add confidence intervals for AUROC and PR AUC using bootstrap or a similar resampling method. This would make comparisons between baselines more defensible. `baselines/utils/run_baseline.py` is the best home for this.
- **Ensemble detector scoring: combine likelihood, rank, logrank, entropy, and perturbation signals into a simple meta-detector. This could be a weighted average or a small logistic model, implemented in `baselines/all_baselines.py`. (Aditya's comment: "I think we proceed like, we create machine generated text with a prompt saying that the generate a human looking text, and then try to compare that machine generated text with the actual human generated text, and if the research results dont hold, then we can try to ensemble the detectors together and see if that gives us a better signal. if the research results hold, we should then test the same signal across different datasets, different base generators, and different prompt styles, then check robustness under length bins, decoding settings, and perturbation strengths. If it still holds, the next step is to quantify which detector contributes most, whether a weighted ensemble improves stability, whether a lightweight meta-classifier can generalize better than any single score, and whether hypothesis testing shows the observed gaps are statistically significant enough to support a blanket ai-vs-human claim rather than a chance fluctuation.")
- Multi-round perturbations: use `n_perturbation_rounds` to perturb text repeatedly and study whether repeated local edits strengthen or destroy the detection signal. That argument is currently parsed in `main.py` but not really used in the main flow.
- Pre-perturbation experiments: make use of `pre_perturb_pct` and `pre_perturb_span_length` to optionally perturb the input before scoring, then compare the resulting detection behavior. This would also belong in `baselines/all_baselines.py` and `baselines/detectGPT.py`.

### Current code gaps that motivate these directions

- `pre_perturb_pct`, `pre_perturb_span_length`, `random_fills`, `random_fills_tokens`, `n_similarity_samples`, and `scoring_model_name` are parsed in `main.py`, but only some of them affect the current execution path.
- The plotting helpers in `baselines/utils/run_baseline.py` depend on outer-scope names and are not fully self-contained.
- The pipeline currently reports mostly point estimates, so it does not yet quantify uncertainty or robustness very well.
- Perturbation logic currently depends on one mask-filling approach, so the repo does not yet answer how much of the signal comes from the perturbation method versus the detector itself.
